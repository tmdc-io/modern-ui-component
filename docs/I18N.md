# ModernUI — i18n Consumer Guide

How internationalization works when you install ModernUI into **your** app with `npx shadcn@latest add …`.

Live demos and API: registry docs site at `/components/i18n` (run `pnpm dev` locally).

---

## Overview

ModernUI i18n is **source you own**, not a runtime npm package:

| Piece | Installed path (typical) | Role |
|-------|--------------------------|------|
| `LanguageProvider` | `hooks/use-translation.tsx` | Holds the active language (`en`, `es`, …) |
| `useTranslation` | same file | Resolves message catalogs for that language |
| `LanguageSelector` | `components/ui/language-selector.tsx` | Optional UI to switch language |

Blocks that need labels ship **`en` + `es` message catalogs inside the block file**. They call `useTranslation(…Messages)` and follow whichever language `LanguageProvider` has set.

```txt
LanguageProvider  ──►  active language (en | es)
        │
        ▼
useTranslation(blockMessages)  ──►  t.title, t.searchPlaceholder, …
        │
        ▼
Block UI (LoginForm, Hero, ModelsTable, …)
```

---

## Two patterns

### 1. Built-in messages (i18n blocks)

These depend on the `i18n` registry item. Installing them also installs `hooks/use-translation.tsx` (and usually `language-selector`).

| Block | Notes |
|-------|--------|
| `login-form` | Sign-in labels + validation strings |
| `hero` | Follow / Explore / Jump to / quality labels |
| `quality-summary-card` | Card chrome strings |
| `data-product-table` | Sort / filter / empty labels |
| `models-table` | Table chrome, search, status labels |
| `model-health-runs` | Legend / panel labels |
| `run-duration` | Chart legend / panel labels |
| `run-metrics` | Metric label defaults |

You wrap the app in `LanguageProvider` once. Switching language updates those blocks automatically.

### 2. Prop-driven (no i18n required)

Examples: `data-product-card`, `application-header`, `dataos-sidebar`, most UI primitives (`button`, `dialog`, …).

You pass already-translated strings as props:

```tsx
<DataProductCard
  title="Análisis del recorrido del paciente"
  subtitle="Adherencia al tratamiento"
  description="Este producto de datos rastrea…"
  badge={{ label: "Calidad", status: "warn" }}
/>
```

Use your own i18n library (next-intl, react-i18next, etc.) or hardcode locale-specific props. ModernUI `LanguageProvider` is optional for these.

## Install

### Recommended order

```bash
npx shadcn@latest init

npx shadcn@latest add tmdc-io/modern-ui-component/theme
npx shadcn@latest add tmdc-io/modern-ui-component/utils
npx shadcn@latest add tmdc-io/modern-ui-component/i18n

# Then primitives / blocks as needed
npx shadcn@latest add tmdc-io/modern-ui-component/login-form
npx shadcn@latest add tmdc-io/modern-ui-component/hero
```

### Auto-pull

If you only run:

```bash
npx shadcn@latest add tmdc-io/modern-ui-component/login-form
```

shadcn still installs `i18n` because it is listed in that block’s `registryDependencies`. Installing `i18n` yourself first keeps foundation order clear.

### Hosted namespace

```bash
npx shadcn@latest add @modernui/i18n
npx shadcn@latest add @modernui/login-form
```

### Monorepo

Install `i18n` in the shared UI package after ModernUI `init` (theme/utils already present):

```bash
npx shadcn@latest add @modernui/i18n -y -c packages/ui
# or: npx shadcn@latest add tmdc-io/modern-ui-component/i18n -y -c packages/ui
```

Export hooks from the package (example):

```json
{
  "exports": {
    "./hooks/*": "./src/hooks/*.ts",
    "./hooks/*": "./src/hooks/*.tsx"
  }
}
```

Then in the app:

```tsx
import { LanguageProvider } from "@workspace/ui/hooks/use-translation"
```

See [CONSUMER.md](./CONSUMER.md) for full monorepo layout.

## Wire the app (required for i18n blocks)

### 1. Wrap with `LanguageProvider`

Next.js App Router example:

```tsx
// app/providers.tsx
"use client"

import { LanguageProvider } from "@/hooks/use-translation"

export function Providers({ children }: { children: React.ReactNode }) {
  return (
    <LanguageProvider defaultLanguage="en">
      {children}
    </LanguageProvider>
  )
}
```

```tsx
// app/layout.tsx
import { Providers } from "./providers"

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  )
}
```

**Props**

| Prop | Default | Meaning |
|------|---------|---------|
| `defaultLanguage` | `"en"` | Initial language before (or without) storage |
| `storageKey` | `"modernui-language"` | `localStorage` key; pass `null` to disable persistence |

### 2. Optional language switcher

```tsx
import { LanguageSelector } from "@/components/ui/language-selector"

export function AppHeader() {
  return (
    <header className="flex items-center justify-end gap-2">
      <LanguageSelector
        languages={[
          { value: "en", label: "English" },
          { value: "es", label: "Español" },
        ]}
      />
    </header>
  )
}
```

`LanguageSelector` reads/writes the nearest `LanguageProvider`. Place it under the provider.

### 3. Render an i18n block

```tsx
import { LoginForm } from "@/components/blocks/login-form"
// path may be @/components/login-form depending on your components.json

export default function LoginPage() {
  return (
    <main className="flex min-h-svh items-center justify-center p-6">
      <LoginForm />
    </main>
  )
}
```

With language `es`, labels such as title / email / sign-in resolve from the block’s Spanish catalog. No extra props required for those chrome strings.

## End-to-end example (login-form)

```tsx
"use client"

import { LanguageProvider } from "@/hooks/use-translation"
import { LanguageSelector } from "@/components/ui/language-selector"
import { LoginForm } from "@/components/blocks/login-form"

export default function Page() {
  return (
    <LanguageProvider defaultLanguage="en">
      <div className="flex min-h-svh flex-col">
        <header className="flex justify-end border-b p-4">
          <LanguageSelector />
        </header>
        <main className="flex flex-1 items-center justify-center p-6">
          <LoginForm />
        </main>
      </div>
    </LanguageProvider>
  )
}
```

Flow:

1. User picks **Español** → `LanguageProvider` sets language to `es` (and persists it).
2. `LoginForm` calls `useTranslation(loginFormMessages)`.
3. `t.signIn` becomes `"Iniciar sesion"` (or the string shipped in that block’s `es` catalog).

## Using `useTranslation` in your own code

Same hook the blocks use:

```tsx
"use client"

import {
  useTranslation,
  type Translations,
} from "@/hooks/use-translation"

const messages = {
  en: {
    dir: "ltr",
    values: {
      greeting: "Welcome",
      cta: "Continue",
    },
  },
  es: {
    dir: "ltr",
    values: {
      greeting: "Bienvenido",
      cta: "Continuar",
    },
  },
} satisfies Translations

export function WelcomeBanner() {
  const { t, dir, language } = useTranslation(messages)

  return (
    <section dir={dir}>
      <h1>{t.greeting}</h1>
      <button type="button">{t.cta}</button>
      <span className="sr-only">language: {language}</span>
    </section>
  )
}
```

### Catalog shape

```ts
type Translations = Record<
  string, // language code: "en", "es", "ar", …
  {
    dir: "ltr" | "rtl"
    values: Record<string, string>
    locale?: string // optional display/locale override
  }
>
```

### Pinning a language (demos / RTL)

```tsx
// Ignore the site language and always use Arabic for this subtree
const { t, dir } = useTranslation(messages, "ar")
```

Omit the second argument so the hook follows `LanguageProvider`.

### Without a provider

If no `LanguageProvider` is present, `useTranslation` keeps a local language state (defaults to `"en"`). Prefer wrapping the app so all blocks stay in sync.

## Adding another locale (e.g. French)

1. Extend each block’s `*Messages` (or your app catalogs) with an `fr` entry that uses the **same keys** as `en` / `es`.
2. Add French to the selector:

```tsx
<LanguageSelector
  languages={[
    { value: "en", label: "English" },
    { value: "es", label: "Español" },
    { value: "fr", label: "Français" },
  ]}
/>
```

3. Set `defaultLanguage="fr"` or let users switch; persistence uses `storageKey`.

Call sites (`t.title`) do not change when you add locales.

## Mixing ModernUI i18n with next-intl / react-i18next

Common approach:

- Use **next-intl** (or similar) for app routes, marketing copy, and prop-driven blocks.
- Use **ModernUI `LanguageProvider`** for installed i18n blocks, **or** keep one source of truth:

**Option A — sync from your app locale**

```tsx
"use client"

import { useLocale } from "next-intl"
import { LanguageProvider } from "@/hooks/use-translation"

export function ModernUiI18nBridge({
  children,
}: {
  children: React.ReactNode
}) {
  const locale = useLocale() // "en" | "es" | …

  return (
    <LanguageProvider
      key={locale}
      defaultLanguage={locale}
      storageKey={null} // avoid fighting next-intl persistence
    >
      {children}
    </LanguageProvider>
  )
}
```

**Option B — props only**

Skip ModernUI language state for screens you fully control; pass translated strings into prop-driven components.

## Customizing block strings

Blocks ship default catalogs as exported constants (e.g. `loginFormMessages`, `heroMessages`). Because files are copied into your repo, you can:

1. **Edit the `es` / `en` values** in the block file directly, or  
2. **Fork the messages object** and change the block to call `useTranslation(yourMessages)`.

There is no separate JSON locale pack for blocks today — catalogs live next to the component.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|----------------|-----|
| Always English | No `LanguageProvider`, or provider below the block | Wrap layout/providers **above** the block |
| Selector does nothing | Selector outside provider | Move `<LanguageSelector />` under `LanguageProvider` |
| Language resets on refresh | `storageKey={null}` or blocked `localStorage` | Use default storage key; check browser privacy mode |
| RTL demo stuck | `useTranslation(messages, "ar")` pins locale | Omit second arg to follow provider |
| `data-product-card` not translating | Prop-driven block | Pass Spanish props or wrap with your own i18n |
| Duplicate providers | Nested `LanguageProvider` with `storageKey={null}` in a demo | Fine for isolated demos; app shell should have one persistent provider |

## Maintainer automation (registry repo)

Docs-site Spanish (`docsCopy`) and block `en`/`es` catalogs are separate from consumer i18n. After adding English docs copy or block message keys, run:

```bash
pnpm docs:i18n
```

That single command:

1. Translates any missing docs strings (MyMemory → generated map)
2. Verifies docs coverage (`missing === 0`)
3. Lints block message key parity (`en` ↔ `es`)

CI runs the verify half only (no network):

```bash
pnpm docs:i18n:verify
```

| Script | Role |
|--------|------|
| `pnpm docs:i18n` | Full refresh: translate + verify |
| `pnpm docs:i18n:verify` | Coverage check + block key lint |
| `pnpm docs:i18n-extract` | Write `scripts/docs-missing-en.json` |
| `pnpm docs:i18n-translate` | MyMemory EN→ES + merge |
| `pnpm docs:i18n-merge` | Rebuild `app/docs-copy-es.generated.ts` |
| `pnpm docs:i18n-check` | Fail if docs Spanish is incomplete |
| `pnpm docs:i18n-lint-blocks` | Fail if block `en`/`es` keys diverge |

Optional: `DOCS_I18N_TRANSLATE_EMAIL=you@example.com` raises MyMemory quota. Use `pnpm docs:i18n-translate -- --dry-run --limit=20` to smoke-test without writing.
